# MVP 范围评审

本文档是范围评审，不是 PRD。它根据回看价值、实现复杂度和价值可验证性，评审当前 Happen MVP 边界。

来源文档：

- `PROJECT.md`
- `docs/superpowers/specs/2026-06-04-happen-mvp-review-value-design.md`

## 1. MVP 应该有哪些功能

MVP 只应该包含直接支持核心验证问题的功能：

> 在看医生或复诊前，照护者能否用 Happen 在约 2 分钟内回看近期儿童健康事实，并比记忆、纸质日历、备忘录、微信消息或 Excel 更清楚地说明情况？

### 孩子与照护者设置

必须包含：

- 用孩子小名添加孩子。
- 为当前用户设置这个孩子下的照护称呼。
- 当用户可访问多个孩子时，选择当前孩子。
- 邀请另一位照护者加入某个孩子。
- 让被邀请照护者查看和编辑这个孩子的 MVP 健康事实。
- 在有帮助的地方展示创建者或最后编辑者。

理由：

- 共享访问是回看价值假设的一部分，因为记录事实的人和陪孩子看医生的人可能不是同一个人。
- 设置必须轻量，不能变成家庭账号管理。

### 大便记录与回看

必须包含：

- 按日期记录大便。
- 当同一天再次记录大便时，增加该日期次数。
- 编辑某一天的大便次数。
- 为和平时不同的情况添加可选观察标签或备注。
- 编辑或删除大便记录。
- 展示距离最近一次大便已经几天。
- 在有意义时展示今天已记录几次大便。

理由：

- 这是最轻的日常健康锚点。
- 它回答一个具体的家长问题：“距离上次大便已经几天？”

### 病程日记录

必须包含：

- 为每个孩子的每个日期创建或编辑一条病程日记录。
- 日期默认今天，但必须由用户明确保存。
- 支持记录最高体温、用药、症状和备注。
- 支持用药和症状的自由文本输入，或轻量快捷词。
- 保存前要求日期以外至少有一个事实。
- 保留用户原始记录的事实。

理由：

- 病程模型应该是按天记录，而不是医学级分钟时间线。
- 家长需要为看医生还原情况，不是建立临床病历。

### 病程时间线回看

必须包含：

- 按天分组查看病程日记录。
- 默认查看截至今天的最近 7 天。
- 让照护者手动调整日期范围。
- 展示所选范围共几天。
- 展示每天的最高体温、用药、症状和备注。
- 对没有记录的日期展示“无记录”，但不要制造愧疚或焦虑。

理由：

- 时间线是看医生前的核心回看界面。
- 手动日期范围足以产生价值，不需要自动识别病程周期。

### 最小安全与边界文案

必须包含：

- 说明用药快捷词只用于记录已经使用过的药品，不是用药建议。
- 避免诊断、推荐、异常提醒、严重度评分和治疗判断语言。
- 使用具体健康事实语言，而不是泛化的事件语言。

理由：

- MVP 只整理事实，不能暗示医学判断。

## 2. 哪些功能应该删除

如果以下功能出现在计划、原型、导航、待办或界面草稿中，应该从 MVP 范围删除。

### 宽泛儿童事件记录

删除：

- 通用事件创建。
- 面向用户的事件类型。
- 自定义事件分类。
- 通用儿童时间线。
- 里程碑、成长回忆、家庭瞬间或照片。

原因：

- 这些会稀释回看价值验证，让产品和备忘录、相册、通用记录工具竞争。

### 家庭社交与协作功能

删除：

- 家庭群组。
- 聊天。
- 评论。
- 动态流。
- 点赞或表情反应。
- 家庭相册。
- 审批流。
- 角色层级。
- 只读权限开关。
- 复杂权限矩阵。

原因：

- MVP 协作只应该表示共享孩子健康事实访问。
- 社交行为会把产品变成另一个产品，也会带来另一个验证问题。

### 医疗建议或临床判断功能

删除：

- AI 医疗咨询。
- 诊断。
- 治疗建议。
- 药物推荐。
- 剂量计算。
- 药品库。
- 症状严重度评分。
- 异常提醒。
- 面向医生的工作流。

原因：

- 在核心记录和回看价值被证明前，这些会过早增加安全、信任、合规和产品复杂度。

### 高级大便追踪

删除：

- 将大便颜色作为必填字段。
- 将大便量作为必填字段。
- 大便照片。
- 大便照片分析。
- Bristol 大便分型。
- 大便异常解释。
- 长期大便趋势图。

原因：

- MVP 问题是大便间隔和每天次数，不是医学化大便分析。

### 提醒与依赖通知的流程

删除：

- 依赖推送通知的提醒。
- 连续打卡。
- 漏记提醒。
- 定时用药提醒。

原因：

- 微信小程序通知机制不够可靠，MVP 价值不能依赖它。

### 通用育儿或习惯功能

删除：

- 课程平台。
- 课时记录。
- 阅读任务。
- 积分或奖励。
- 运动记录。
- 刷牙记录。
- 通用习惯追踪。
- 通用儿童任务管理。

原因：

- 这些与健康事实回看假设无关。

## 3. 哪些功能会增加复杂度

以下功能看起来接近 MVP，但会显著增加复杂度。除非验证证明必要，否则应该延后或极简实现。

### 多个孩子

增加的复杂度：

- 所有数据访问都必须按孩子隔离。
- 需要孩子切换。
- 每个孩子都有独立空状态。
- 必须避免跨孩子记录泄漏。

结论：

- 保留在 MVP，但用最小方式实现。

原因：

- 家长可能有多个孩子，而且每条健康事实都必须明确属于一个孩子。

### 被邀请照护者

增加的复杂度：

- 邀请链接或微信分享卡片。
- Membership 记录。
- 按孩子授权。
- 并发编辑边界情况。
- 创建者和最后编辑者展示。

结论：

- 保留在 MVP，但只做简单的被邀请照护者可查看和编辑。

原因：

- 共享访问直接对应“记录的人可能不是看医生的人”的问题。

### 大便观察标签

增加的复杂度：

- 标签词表。
- 标签编辑。
- 防止标签被理解为医学分型。

结论：

- 只保留可选轻量标签和备注。

原因：

- 对异常日有帮助，但不是回答大便间隔问题的必要条件。

### 用药和症状快捷词

增加的复杂度：

- 可配置词表。
- 安全文案。
- 用户可能把快捷词误解为推荐。

结论：

- 只有在明确作为记录快捷入口时才保留，不能作为医疗建议。

原因：

- 它们能降低病程日记录的输入成本。

### 病程时间线日期范围

增加的复杂度：

- 日期范围控件。
- 空日期展示。
- 所选天数计算。
- 从回看界面编辑记录。

结论：

- 保留在 MVP。

原因：

- 这是看医生前回看价值的核心界面。

### 自动识别病程周期

增加的复杂度：

- 推断规则。
- 症状反复出现时的边界情况。
- 系统猜错时的信任问题。
- 更复杂的数据模型。

结论：

- 延后。

原因：

- 手动日期范围已经足以验证回看价值。

### 图表与分析

增加的复杂度：

- 聚合逻辑。
- 可视化设计。
- 缺失数据边界情况。
- 产品变成仪表盘的风险。

结论：

- 延后。

原因：

- MVP 统计只应保留距离上次大便几天、当日大便次数和病程时间线所选天数。

## 4. 哪些功能无法验证价值

以下功能可能带来活跃、完整感或视觉打磨，但不能直接验证 Happen 是否比记忆、纸质日历、备忘录、微信消息或 Excel 更适合近期儿童健康事实回看。

### 成长相册或回忆时间线

为什么无法验证 MVP 价值：

- 它验证的是情感分享和记忆保存，不是看医生前健康事实回看。

### 家庭聊天、评论或动态

为什么无法验证 MVP 价值：

- 使用量只能说明社交互动，不说明事实回看是否更快、更清楚。

### 积分、奖励或连续打卡

为什么无法验证 MVP 价值：

- 它们可能提高记录行为，但不能证明记录在需要时真的有用。

### 长期分析仪表盘

为什么无法验证 MVP 价值：

- 它验证的是分析兴趣，不是 2 分钟近期回看场景。

### AI 总结

为什么无法验证 MVP 价值：

- 它可能掩盖底层记录和时间线结构是否已经有效。
- 它会过早引入信任和安全问题。

### 通知和提醒

为什么无法验证 MVP 价值：

- 提醒成功验证的是通知机制和习惯养成，不是已记录事实是否更容易回看。

### 详细医学结构化

为什么无法验证 MVP 价值：

- 药品库、剂量字段、严重度评分和精确时间要求验证的是临床数据完整性，不是轻量家长回看。

## 推荐 MVP

推荐 MVP 是：

> 一个微信小程序，用于记录和回看少量共享的儿童健康事实：大便间隔/次数和按天记录的病程事实，并配套最小孩子/照护者设置与被邀请照护者访问。

### MVP 必须包含

- 首次使用时设置孩子小名和照护称呼。
- 多孩子用户的当前孩子选择器。
- 孩子设置中的照护者列表和邀请操作。
- 按孩子维度的照护者 Membership。
- 简单的被邀请照护者加入流程。
- 被邀请照护者可共享查看和编辑。
- 每个孩子每天一条大便记录。
- “记一次大便”快捷操作，在确认后递增所选日期次数。
- 大便记录编辑和删除。
- 可选大便观察标签和备注。
- 首页展示距离上次大便几天。
- 在有意义时展示今天已记录几次大便。
- 每个孩子每天一条病程日记录。
- 病程日字段：最高体温、用药、症状和备注。
- 用药和症状自由文本输入。
- 可选用药和症状快捷词，并配安全文案。
- 每条健康事实都必须显式保存。
- 病程日记录保存时，要求日期以外至少有一个事实。
- 按天分组的病程时间线。
- 病程时间线默认最近 7 天。
- 手动调整日期范围。
- 时间线展示每天的最高体温、用药、症状和备注。
- 在有帮助的地方展示创建者或最后编辑者。

### MVP 应该排除

- 通用事件记录器。
- 成长相册。
- 照片。
- 家庭社交功能。
- 聊天、评论、动态、点赞或表情反应。
- 复杂角色、只读权限、审批流或家庭群组。
- 医疗建议、诊断、治疗建议、药物推荐或剂量计算。
- 药品库。
- 自动识别病程周期。
- 大便分型、大便照片分析或异常提醒。
- 依赖通知的提醒。
- 图表、趋势和仪表盘分析。
- 通用习惯、任务、课程、积分、阅读、运动或刷牙。
- 面向医生或老师的工作流。

### 验证门槛

只有当 MVP 证明以下信号时，才应认为它有价值：

- 家长确实有近期记录或回看孩子发热、用药、病程或大便事实的具体困难。
- 至少一部分目标家长愿意尝试这样一个很窄的工具。
- 照护者可以在几秒内记录一次大便。
- 照护者可以记录或更新病程日记录，而不需要填写临床级时间线。
- 照护者可以在约 2 分钟内回看所选日期范围，并更清楚地说明孩子近期病程。
- 当一个照护者记录事实、另一个照护者需要回看时，共享访问确实有帮助。

### 产品管理建议

先构建能够验证回看价值的最小版本：

1. 先做孩子设置、大便状态和大便记录。
2. 再做病程日记录和 7 天时间线。
3. 单用户回看闭环可用后，再加入被邀请照护者访问。
4. 在真实家长测试证明窄范围回看闭环有价值前，不添加宽泛育儿、医疗、社交、分析或提醒功能。
