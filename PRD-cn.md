# Happen MVP PRD

本文档定义 Happen 的第一版 MVP。Happen 是面向 0-12 岁儿童家长和被邀请照护者的微信小程序。MVP 保持克制：只记录和回看少量儿童健康事实，帮助照护者在看医生前或看医生时回答实际问题。

来源文档：

- `PROJECT.md`
- `docs/superpowers/specs/2026-06-04-happen-mvp-review-value-design.md`

## Product Goals

1. 帮助照护者在真实照护场景中快速记录儿童健康事实。
2. 帮助照护者在约 2 分钟内回看近期大便和病程事实。
3. 帮助陪孩子看医生的照护者访问另一位照护者记录的事实。
4. 验证一个很窄的健康事实回看工具，是否比记忆、纸质日历、备忘录、微信消息或 Excel 更有用。
5. 保持 MVP 聚焦事实回看，不做医疗建议、家庭社交、通用事件记录或长期分析。

成功信号：

- 在 5 位目标家长访谈中，至少 3 位描述过近期记录或回看发热、用药、大便或类似儿童健康事实的具体困难。
- 至少 2 位愿意尝试一个只关注大便间隔和病程日记录的工具。
- 照护者可以在几秒内记录一次大便。
- 照护者可以创建或更新病程日记录，而不需要精确医学时间线。
- 照护者可以在约 2 分钟内回看所选日期范围，并更清楚地说明孩子近期病程。
- 在共同照护场景中，一位照护者记录事实，另一位照护者可以在看医生前或看医生时回看。

## Personas

### 主要家长

- 0-12 岁儿童家长。
- 经常使用记忆、纸质日历、备忘录、微信消息或 Excel 记录健康事实。
- 看医生时需要回答具体问题，例如发热从什么时候开始、最高体温、吃过什么药、有什么症状、上次大便是什么时候。

### 陪诊照护者

- 可能是家长，也可能是其他家庭照护者。
- 不一定是最初记录事实的人。
- 需要在门诊前或门诊中快速回看近期健康事实。
- 比起复杂录入，更重视清楚的按天回看。

### 被邀请照护者

- 帮忙记录或回看同一个孩子健康事实的祖辈或其他家庭照护者。
- 可能不熟悉复杂应用。
- 需要简单的按孩子访问能力，不需要家庭社交或权限管理。

## User Stories

1. 作为准备在下一次健康事件发生前先建好记录入口的家长，我想用孩子小名添加孩子，并设置自己的照护称呼，这样之后记录会归到正确孩子名下，其他照护者也能看懂是谁记录的，而不需要填写正式身份信息。
2. 作为需要照护多个孩子的照护者，我想在记录或回看前清楚切换当前孩子，这样在匆忙照护时不会把兄弟姐妹的健康事实混在一起。
3. 作为在家记录事实但不一定亲自陪孩子看医生的家长，我想邀请另一位照护者加入这个孩子的记录，这样对方需要向医生说明情况时，可以回看同一份事实。
4. 作为担心孩子大便间隔变长的照护者，我想按日期快速记录每一次大便，这样之后可以回答距离上次大便是 2 天、3 天还是 4 天。
5. 作为发现漏记或记错日期的照护者，我想编辑或删除某一天的大便记录，这样之后展示的大便间隔是可信的。
6. 作为正在处理发热、用药或症状的照护者，我想记录当天重要事实，这样到门诊时不用靠记忆还原。
7. 作为疲惫、忙碌或只方便用手机记录病程的照护者，我想用快捷词和自由文本记录症状与用药，这样不用填写临床级时间线，也能留下足够有用的信息。
8. 作为准备看医生或复诊的照护者，我想按所选日期范围回看病程事实，这样可以按天清楚说明发生了什么。
9. 作为回看别人记录内容的照护者，我想在有帮助的地方看到创建者或最后编辑者，这样当某条事实看起来陌生时，可以知道是谁最后处理过。
10. 作为使用 Happen 记录儿童健康事实的照护者，我希望产品避免诊断和用药建议，这样我可以把它当作事实记录工具，而不会误以为它替代医生。

## Functional Requirements

### 功能 1：首次使用的孩子与照护者设置

#### 用户价值

照护者可以用最少步骤开始使用 Happen，不需要填写不必要的敏感信息。产品会立刻把所有记录绑定到一个孩子和一个照护者称呼。

#### 业务规则

- 如果用户还没有孩子，展示首次使用设置。
- 设置必须包含孩子小名。
- 设置必须包含当前用户在该孩子下的照护称呼。
- 建议称呼包括妈妈、爸爸、奶奶、爷爷和其他。
- MVP 不要求孩子真实姓名、生日、性别、头像或照护者真实姓名。
- 设置完成后，用户成为该孩子的照护者成员，并进入该孩子首页。
- 照护称呼只是展示文本，不是权限角色，也不要求唯一。

#### 异常场景

- 如果孩子小名为空，阻止完成并提示用户填写小名。
- 如果照护称呼为空，阻止完成并提示用户选择或填写称呼。
- 如果设置因网络或服务错误失败，保留已输入内容并允许重试。
- 如果同一用户已有同名孩子，允许创建，但孩子选择器必须足够清楚，避免误切换。

#### 验收标准

- 给定一个没有孩子的新用户，当打开 Happen 时，应看到首次使用设置。
- 给定用户输入孩子小名和照护称呼，当确认时，应创建孩子和成员关系。
- 给定设置成功，当用户进入首页时，新创建的孩子应被选中。
- 给定设置页被打开，当用户未确认时，不应创建孩子或健康事实。
- 给定孩子小名为空，当用户确认设置时，应阻止设置完成，并且不创建孩子。
- 给定照护称呼为空，当用户确认设置时，应阻止设置完成，并且不创建成员关系。
- 给定设置提交失败，当展示错误时，之前输入的孩子小名和照护称呼应保留，可用于重试。

### 功能 2：当前孩子上下文与首页

#### 用户价值

照护者可以看到当前选中的孩子，并立即访问最重要的记录和回看操作。

#### 业务规则

- 首页始终运行在一个已选孩子上下文中。
- 如果用户可访问多个孩子，提供孩子选择器。
- 所有首页状态、操作和记录都必须只属于当前选中的孩子。
- 首页把大便状态作为主要日常健康锚点。
- 首页提供记一次大便、查看或编辑大便记录、记录今天病程、查看病程时间线等操作。
- 首页必须使用具体健康事实语言，不使用通用事件语言。

#### 异常场景

- 如果当前选中孩子不可访问，提示用户选择另一个可访问孩子。
- 如果用户没有任何可访问孩子，回到首次使用设置或邀请入口。
- 如果状态数据加载失败，展示重试状态，不创建记录。

#### 验收标准

- 给定照护者只有一个孩子，当首页加载时，该孩子应自动选中。
- 给定照护者有多个孩子，当切换孩子时，所有展示状态和操作应更新到该孩子。
- 给定首页加载，若已有大便记录，应展示距离上次大便几天。
- 给定该孩子没有大便记录，应展示空状态，而不是错误间隔。
- 给定照护者从孩子 A 切换到孩子 B，当记录健康事实时，新事实只属于孩子 B。
- 给定用户失去当前选中孩子的访问权限，当首页重新加载时，不应展示该孩子记录，并应要求用户选择另一个可访问孩子。
- 给定首页状态加载失败，当展示重试状态时，不应因为加载失败而创建任何健康事实。

### 功能 3：邀请照护者与共享成员关系

#### 用户价值

记录健康事实的人和陪孩子看医生的人都可以访问同一个孩子的记录。

#### 业务规则

- 照护者可以在孩子设置中邀请另一位照护者加入特定孩子。
- 邀请只作用于一个孩子。
- 每个邀请保存目标孩子、邀请人、创建时间、过期时间和状态。
- MVP 邀请链接默认创建后 7 天过期，除非实现出于安全考虑选择更短有效期。
- 只有当邀请 token 存在、格式正确、未过期、未撤销或禁用，并且指向一个仍然存在且邀请人仍有访问权限的孩子时，邀请才有效。
- 当当前时间晚于邀请过期时间时，邀请视为已过期。
- MVP 邀请在过期前可以被多个用户打开，因为微信分享卡片或链接不绑定已知接收人。
- 同一用户多次接受同一邀请必须是幂等的：不得创建重复成员关系。
- 被邀请用户必须确认加入，并选择照护称呼。
- 加入后，被邀请照护者可以查看和编辑该孩子的 MVP 健康事实。
- MVP 权限保持简单：一个孩子下所有照护者都可以查看和编辑该孩子的大便记录、病程日记录和时间线。
- MVP 不包含家庭群组、角色层级、只读权限、审批流、聊天、评论或动态。
- 记录保存创建者和最后编辑者。

#### 异常场景

- 如果邀请 token 缺失、格式错误、不存在、已撤销、已禁用、指向已删除孩子，或指向邀请人已无权访问的孩子，展示无效邀请状态，不创建成员关系。
- 如果邀请在过期时间之后被打开，展示已过期邀请状态，不创建成员关系。
- 如果被邀请用户加入前取消，不创建成员关系。
- 如果被邀请用户已经有该孩子访问权限，不创建另一个成员关系，直接进入该孩子首页。
- 如果同一用户多次确认同一邀请，第一次成功确认创建成员关系，后续确认视为已加入。
- 如果两个不同用户在过期前打开同一有效邀请，两人都可以用各自照护称呼加入同一个孩子。
- 如果成员关系创建失败，保留确认页并允许重试。

#### 验收标准

- 给定已有照护者打开孩子设置，当选择邀请照护者时，Happen 应提供按孩子限定的邀请。
- 给定被邀请用户打开有效邀请，当选择照护称呼并确认时，应为该孩子创建成员关系。
- 给定被邀请照护者加入成功，当打开首页时，被邀请孩子应被选中或出现在孩子选择器中。
- 给定照护者只属于某个孩子，当请求记录时，只能访问有成员关系的孩子。
- 给定邀请 token 缺失、格式错误、不存在、已撤销、已禁用、指向已删除孩子，或指向邀请人已无权访问的孩子，当用户打开它时，Happen 应展示无效邀请状态，并且不创建成员关系。
- 给定邀请已超过过期时间，当用户尝试加入时，Happen 应展示已过期邀请状态，并且不创建成员关系。
- 给定被邀请用户在确认页取消，当离开流程时，不应创建成员关系。
- 给定已经属于该孩子的用户再次打开同一邀请，当确认或继续时，Happen 不应创建重复成员关系，并应进入该孩子首页。
- 给定同一用户提交同一有效邀请两次，当两次提交都被处理后，该用户在该孩子下应只有一个成员关系。
- 给定两个不同用户在过期前打开同一有效邀请，当各自选择照护称呼并确认时，两人都应获得该孩子的独立成员关系。
- 给定被邀请照护者通过邀请加入孩子 A，当请求没有孩子 B 成员关系的记录时，应被拒绝访问。

### 功能 4：孩子设置

#### 用户价值

照护者可以管理共享健康事实所需的最少信息，而不会进入复杂家庭管理。

#### 业务规则

- 孩子设置展示孩子小名。
- 孩子设置展示当前用户在该孩子下的照护称呼。
- 孩子设置按显示称呼展示该孩子的照护者列表。
- 孩子设置提供邀请照护者操作。
- 当前用户可以编辑自己的照护称呼。
- 照护称呼不要求唯一，也不定义权限等级。
- 孩子设置不得暴露家庭群组、角色、只读开关、审批、聊天或动态。

#### 异常场景

- 如果小名更新失败，保留原小名并展示重试。
- 如果照护称呼更新失败，保留原称呼并展示重试。
- 如果照护者列表加载失败，展示重试状态，并尽可能保留其他设置。

#### 验收标准

- 给定照护者打开孩子设置，应看到孩子小名、自己的称呼、照护者列表和邀请操作。
- 给定照护者更新自己的称呼，当更新成功后，后续创建者和最后编辑者展示应使用新称呼。
- 给定两个照护者使用同一个称呼，应允许保存。
- 给定照护者打开设置，不应看到社交、聊天、角色层级或审批控件。
- 给定孩子小名更新失败，当展示错误时，旧小名仍应生效。
- 给定照护称呼更新失败，当展示错误时，旧称呼仍应生效。
- 给定照护者列表加载失败，当展示设置页时，不应把任何照护者作为持久化变更移除或隐藏。

### 功能 5：大便记录与回看

#### 用户价值

照护者可以快速回答孩子距离上次大便已经几天，以及某个日期记录了几次大便。

#### 业务规则

- 每个孩子每天最多有一条大便记录。
- 大便记录保存日期、次数、可选观察标签、可选备注、创建者和最后编辑者。
- 首页的“记一次大便”打开轻量确认面板。
- 日期默认今天，并可在确认前修改。
- 确认后，将当前孩子在所选日期的大便次数加 1。
- 打开应用或打开确认面板都不能创建记录。
- 照护者可以编辑日期、次数、观察标签和备注。
- 照护者可以删除大便记录。
- 可选观察标签只是异常日快捷记录词，不是大便医学分型。
- MVP 不包含大便颜色、照片、Bristol 分型、异常提醒或趋势图。

#### 异常场景

- 如果用户取消确认面板，不创建或修改记录。
- 如果所选日期已有记录，确认后增加现有记录次数，而不是创建重复日期记录。
- 如果编辑后的次数为 0 或为空，要求修正或提示删除。
- 如果保存失败，保留用户输入并展示重试。
- 如果两位照护者更新同一天记录，系统必须保留有效次数和最后编辑者信息。

#### 验收标准

- 给定孩子今天没有大便记录，当照护者确认“记一次大便”时，今天次数应变为 1。
- 给定孩子今天已有 1 次，当照护者再次确认今天记录时，今天次数应变为 2。
- 给定确认面板被打开，当照护者未确认就关闭时，次数不应变化。
- 给定已有大便记录，当照护者编辑次数、标签或备注并保存时，应更新该孩子该日期记录。
- 给定大便记录被删除，当首页重新计算状态时，应使用下一条最近记录或展示空状态。
- 给定应用被打开，当用户没有执行记录操作时，不应创建大便记录。
- 给定确认面板被打开，当用户没有点击确认时，不应创建大便记录。
- 给定用户把次数编辑为 0 或空，当尝试保存时，应阻止保存，或提示用户改为删除记录。
- 给定所选日期已有记录，当用户记一次大便时，Happen 应更新现有日期记录，而不是为同一孩子同一日期创建重复记录。
- 给定保存失败，当展示错误时，之前已保存记录应保持不变，用户未保存输入应可用于重试。

### 功能 6：病程日记录

#### 用户价值

照护者可以记录看医生可能需要的按天事实，而不需要填写医学级时间线。

#### 业务规则

- 每个孩子每天最多有一条病程日记录。
- 病程日记录保存日期、最高体温、用药、症状、备注、创建者和最后编辑者。
- “记录今天病程”打开日期默认今天的表单。
- 如果该孩子该日期已有记录，表单编辑现有记录。
- 表单只能在用户明确确认后保存。
- 日期必填。
- 保存前必须至少有一个日期以外的事实。
- 最高体温、用药、症状和备注分别可选。
- 用药和症状支持自由文本。
- 症状快捷词可以作为记录快捷入口。
- 用药快捷词只能作为已使用药品的记录入口。
- 用药快捷词必须配安全文案，说明不是用药建议。
- MVP 不包含剂量计算、药品说明、药品库、症状严重度评分或医疗建议。

#### 异常场景

- 如果用户只填写日期就尝试保存，阻止保存并提示写一点之后想回看的情况。
- 如果用户取消表单，不创建或修改记录。
- 如果日期改为已有记录的日期，加载该记录进行编辑，或在合并前清晰提醒。
- 如果保存失败，保留已输入内容并展示重试。
- 如果两位照护者编辑同一天病程记录，保留有效的最新版本和最后编辑者信息。

#### 验收标准

- 给定照护者打开病程日表单，当未保存时，不应创建记录。
- 给定照护者只输入症状并保存，病程日记录应有效。
- 给定照护者只输入用药文本并保存，病程日记录应有效。
- 给定照护者只输入备注并保存，病程日记录应有效。
- 给定照护者只填写日期并尝试保存，应阻止保存。
- 给定所选日期已有记录，当照护者打开表单时，应编辑现有记录，而不是创建重复记录。
- 给定照护者取消表单，当返回首页时，不应创建或修改病程日记录。
- 给定展示用药快捷词，当用户查看时，应同时展示安全文案，说明它们只记录已使用药品，不是建议。
- 给定日期被改为另一个已有记录的日期，当选择该日期时，Happen 不得静默覆盖任何一条记录。
- 给定保存失败，当展示错误时，最后一次已保存记录应保持不变，用户未保存输入应可用于重试。

### 功能 7：病程时间线回看

#### 用户价值

照护者可以按天回看近期病程事实，在看医生前或看医生时更清楚地说明情况。

#### 业务规则

- 时间线限定在当前选中的孩子。
- 时间线使用所选日期范围，不存储或推断病程周期。
- 默认日期范围为截至今天的最近 7 天。
- 照护者可以手动修改开始日期和结束日期。
- 时间线展示所选范围共几天。
- 每个日期展示已有的最高体温、用药、症状和备注。
- 没有病程日记录的日期可展示“无记录”，但不能使用制造愧疚的语言。
- 时间线可在有帮助的地方展示创建者或最后编辑者。
- 时间线必须避免“自动识别本次病程”“系统判断病程”等文案。
- 时间线不能看起来像诊断报告。

#### 异常场景

- 如果所选范围没有病程日记录，展示空状态，并提供记录今天病程或修改日期范围操作。
- 如果开始日期晚于结束日期，阻止应用并提示修正。
- 如果时间线数据加载失败，展示重试，不修改已存记录。
- 如果某一天只有部分数据，展示已有事实并省略空字段。

#### 验收标准

- 给定照护者打开病程时间线，默认范围应为截至今天的最近 7 天。
- 给定照护者修改日期范围，时间线应更新到所选范围，并展示正确天数。
- 给定范围内存在病程日记录，应按日期分组展示记录事实。
- 给定范围内没有病程日记录，应展示空状态，且不创建记录。
- 给定某一天没有记录，时间线可展示无记录，但不暗示照护者漏记。
- 给定开始日期晚于结束日期，当照护者应用范围时，应拒绝该范围，并保留之前有效的时间线。
- 给定时间线数据加载失败，当展示重试状态时，不应创建、编辑或删除任何病程日记录。
- 给定某一天只有症状，没有用药或体温，当时间线渲染该日期时，应展示症状并省略空字段。
- 给定时间线打开，当展示记录时，页面不得声称自动识别了病程周期。

### 功能 8：安全与范围边界

#### 用户价值

照护者可以把 Happen 当作事实记录工具使用，而不会误以为它提供医疗建议或临床判断。

#### 业务规则

- 产品只记录事实和用户输入的观察。
- 产品不得诊断、推荐药物、计算剂量、判断是否就医或替代医生。
- 用药快捷词只是记录快捷入口。
- 用户界面应使用具体健康事实语言。
- 用户界面不应把通用“事件”作为主概念。
- MVP 不包含 AI 总结、通知、图表、医疗推荐逻辑、复杂权限、家庭社交功能或存储式病程自动识别。

#### 异常场景

- 如果用户在备注中输入类似医疗建议的内容，只作为用户输入文本保存，不展示为系统指导。
- 如果展示用药快捷词，也必须在相关位置展示安全文案。
- 如果某个功能请求需要诊断、推荐、提醒或分析，应标记为未来范围或范围外，而不是加入 MVP。

#### 验收标准

- 给定用户查看用药快捷词，界面应说明它们只用于记录已经使用过的药品或护理用品，不是建议。
- 给定用户查看时间线，页面应按日期展示事实，不提供诊断或治疗建议。
- 给定用户浏览 MVP，不应出现家庭社交、通用事件、提醒、图表、AI 总结、药品库或医疗建议功能。
- 给定健康事实被打开或编辑，只有用户明确操作后才保存。
- 给定用户在备注中输入类似医疗建议的文本，当之后展示时，应作为用户输入备注展示，而不是系统指导。
- 给定用户选择症状，当保存或回看记录时，Happen 不应计算严重程度或推荐药物。
- 给定有人请求提醒、图表、AI 总结或药品库等未来范围功能，当评估 MVP 范围时，不应把它视为 MVP 需求。

## User Flow

### 首次使用流程

1. 用户打开 Happen。
2. 如果用户没有孩子，Happen 展示设置页。
3. 用户输入孩子小名。
4. 用户选择或输入照护称呼。
5. 用户确认。
6. Happen 创建孩子和成员关系。
7. 用户进入该孩子首页。

### 日常大便记录流程

1. 照护者打开当前孩子首页。
2. 照护者查看大便状态。
3. 照护者点击“记一次大便”。
4. 确认面板打开，日期默认今天。
5. 照护者确认，或修改日期后确认。
6. Happen 增加该孩子该日期的大便次数。
7. 首页更新距离上次大便几天和今天次数。

### 病程日记录流程

1. 照护者打开当前孩子首页。
2. 照护者点击“记录今天病程”。
3. 病程日表单打开，日期默认今天。
4. 照护者输入一个或多个事实：最高体温、用药、症状或备注。
5. 照护者保存。
6. Happen 创建或更新该孩子该日期的病程日记录。

### 看医生前回看流程

1. 照护者打开当前孩子首页。
2. 照护者点击“查看病程时间线”。
3. 时间线打开，默认最近 7 天。
4. 照护者按需要调整日期范围。
5. 照护者回看每天的最高体温、用药、症状和备注。
6. 照护者使用时间线向医生或其他照护者说明近期事实。

### 共同照护流程

1. 已有照护者打开孩子设置。
2. 已有照护者选择邀请照护者。
3. Happen 生成按孩子限定的微信分享卡片或邀请链接。
4. 被邀请用户打开邀请。
5. 被邀请用户确认孩子和访问含义。
6. 被邀请用户选择照护称呼。
7. 被邀请用户加入并进入该孩子首页。
8. 被邀请用户可以查看和编辑该孩子的 MVP 健康事实。

## Acceptance Criteria

MVP 级验收：

1. 新照护者可以用孩子小名和照护称呼创建孩子。
2. 照护者可以在可访问孩子之间切换，且不会误看或误编辑另一个孩子的记录。
3. 照护者可以邀请另一位照护者加入一个孩子。
4. 被邀请照护者可以加入，并且只能查看或编辑该孩子的 MVP 健康事实。
5. 照护者可以通过明确确认记录一次大便。
6. 同一天重复记录大便时，应递增同一条日期级记录。
7. 照护者可以编辑和删除大便记录。
8. 当有数据时，首页应准确展示距离上次大便几天。
9. 照护者可以用日期以外至少一个事实创建或更新病程日记录。
10. 打开病程表单本身永远不会创建记录。
11. 照护者可以按默认 7 天范围回看病程日记录。
12. 照护者可以手动调整病程时间线日期范围。
13. 时间线按天展示记录事实，不诊断，也不推荐治疗。
14. 用药快捷词必须明确呈现为记录快捷入口，而不是建议。
15. MVP 不包含家庭社交、通用事件记录器、药品库、AI 总结、依赖通知的提醒、图表或自动识别病程周期。

回看价值验收：

1. 照护者可以在几秒内记录一次大便。
2. 照护者可以记录病程日，而不需要填写精确医学时间线。
3. 照护者可以用所选日期范围时间线，在约 2 分钟内说明近期病程事实。
4. 共同照护场景可用：一位照护者记录事实，另一位照护者可以回看。

## Future Scope

只有在 MVP 通过真实家长验证窄范围回看价值后，才考虑未来范围。

可能的未来功能：

- 对用户已记录事实做 AI 总结，但只限总结，不能提供医疗建议。
- 不让 MVP 价值依赖不稳定推送通知的提醒能力。
- 更可配置的症状和用药记录词。
- 更好的共同照护编辑历史。
- 面向照护者的事实摘要导出或分享。
- 如果家长证明近期看医生场景之外也有反复回看价值，再考虑轻量趋势视图。

明确不属于 MVP：

- 医疗建议、诊断、治疗建议、药物推荐或剂量计算。
- 药品库。
- 自动识别病程周期。
- 大便分型、大便照片分析或异常提醒。
- 成长相册、照片、家庭聊天、评论、动态或社交功能。
- 通用习惯、课程、任务、积分、阅读、运动或刷牙。
- 面向医生或老师的工作流。
